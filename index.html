Let's Try to solve

Question - Given a sorted array of elements, and an input, find whether the given input element exists in the array. If yes, print it's position, otherwise print -1

Step 1: Understand and Analyze the problem

Ask Yourself -

  • Are you able to understand the question fully?
  • Can you restate the problem in your own words
  • Would you be able to explain the question to a layman?
  • What and how many inputs are required?
  • What would be the output for those inputs
  • Can you separate out some modules or parts from the problem?
  • Do you have enough information?

Step 2: Go through the sample inputs and examples

The 4 steps -

  • Take very simple examples and find the output
  • Take more complex and bigger inputs to see what will be the output, how many use cases do we want
  • Try out the problem with no input, what should be the output now
  • Try out the problem with invalid input, what should be the output now

Step 3: Break down the problem

Write down somewhere, and if the problem is easy, think of the steps you want to take in mind

Step 4: Start solving/coding

Keep these things in mind:

  • The point where you started
  • Where are you right now?
  • What is your destination?

Step 5: Look Back and improve

Don‚Äôt just give up when you are done, give up when you feel content with the possible solution(s) and you‚Äôve explored the problem completely!

Ask yourself:

  • Does this code run for every possible input (including the edge cases)
  • Is there any other way to solve the problem?
  • Is the code efficient? Can it be more efficient?
  • Is the code readable?
  • If someone else showed you this code, would you be able to understand it?
  • Can the performance be improved?
  • Can some other algorithm be used which provides better results?

The Big O?

Motivation?

  • We want to find out which algorithm is the best
  • But we can't compare actual runtimes
  • Actual runtimes, vary from machine to machine
  • And even on the same machine, the runtime might vary from time to time
  • Also, for very fast machinies, time measurement might not be precise enough
  • Big O provides a general trend on how the algorithm proceeds with respect to the input
  • This helps us find the parts of code that are less efficient

Apart from these, the greatest motivation for you to know well enough about the big O would be the fact that it comes in job interviews a lot üí∞üè¢

The Big O, finally üòå

üëâ Maybe a rough representation of the growth trends of an algorithm?

üëâ We suppose that each simple operation (like arithmetic, or assignment) takes a constant time, and hence, with the increase of number of operations, the runtime of an algorithm will also increase!

üëâ We just need to find out general trends, Big O is the upper bound.

üëâ Big O allows us to formally represent this fuzzy growth

Formal Definition To Banti hai boss üòÖ

We say that an algorithm is O(f(n)), if the number of simple operations the computer has to do is less than any scalar (constant) times f(n), as n increases

  • f(n) can be constant - O(1)
  • f(n) can be linear - O(n)
  • f(n) can be quadratic - O(n^2)
  • f(n) can be polynomial of power x - O(n^x)
  • f(n) can be logarithmic - O(log(n))
  • and so on...

Some Common Runtimes

Let's discuss some common runtimes

1. Constant Runtime

  • Represented by O(1)
  • No matter what is the input value, and/or how many inputs or elements are dealt with, the algorithm will always take the same amount of time to execute
  • This can be made possible if we are able to find out or derive a formula using which we can calculate the result directly using mathematical computation rather than iterating over the collection of data.
  • For example, to calculate the sum of the first N natural numbers, we can either run a loop from 1 to N (Runtime‚Ää‚Äî‚ÄäO(N)), or we can directly use the formula, n(n+1)/2 (Runtime‚Ää‚Äî‚ÄäO(1))

Constant Runtime

2. Logarithmic Runtime

  • Represented by O(log(n))
  • Doubling the number of elements/inputs, the amount of processing that has to be done does not get doubled. In fact, the amount of processing to be done gets ‚Äòx‚Äô times (approximate) when we increase the elements/inputs, 2^x times
  • This generally occurs when we compute the result by iterating over only some of the elements in the given collection of data rather than iterating through each of the elements
  • A very good example of this algorithm is the Binary Search
  • You might have guessed, the graph for this would be a logarithmic graph.

while (x>0) {
  x = x/2;
}

Logarithmic Runtime

3. Linear Runtime

  • Represented by O(n)
  • As the name suggests, this represents a linear relationship. As we double the number of elements or inputs, the amount of processing that needs to be done also increases
  • This occurs when we need to iterate over each element of a collection of data
  • A very simple example for this is to add the elements in an array. To do so, we will have to iterate over each element once, which would mean n iterations.

for (i=0; i≺; i++) {
  sum += arr[i];
}

Linear Runtime

4. Quasilinear Runtime

  • Represented as O(n*log(n))
  • This is less efficient than linear runtime but still, it is considered to be fairly good and is majorly found in the divide and conquer algorithms
  • A very simple explanation for this is that the algorithm performs ‚ÄòN‚Äô number of operations and each operation runs in O(log(n)) (logarithmic) time
  • The famous Merge Sort algorithm runs in O(n*log(n)) time

Quasilinear Runtime

5. Quadratic Runtime

  • Expressed as O(n^2)
  • Till now you might have guessed, in this case, increasing the elements or the input by x times would
  • This generally happens when each element in a collection has to be compared with every other element
  • A very common example for this would be 2-dimensional array traversal

for (i=0; i≺row; i++) {
  for (j=0; i≺col; i++) {
    print (arr[i][j])
  }
}

Quadratic Runtime

6. Exponential Runtime

  • Represented as O(2^n)
  • In this case, the addition of a single element in the collection, or incrementing input by 1 would cause the amount of processing required to be doubled.
  • If you end up solving a problem with this time complexity, you might want to look back and try to improve the solution. The algorithm having exponential runtime is going to be very slow
  • As the name suggests, the graph follows an exponential trend
